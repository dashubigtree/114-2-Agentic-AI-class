### 1.2 完整請求資料流（一次「執行分析」）

```
使用者貼入郵件 → [Streamlit] 點擊「執行分析」
    │
    ▼ HTTP POST /api/v1/analyze
[Flask app.py]
    │
    ├─ Layer 1：extract_features(email_content)
    │     └─ 計算 76 維特徵向量（手工特徵 + TF-IDF）
    │
    ├─ Layer 2：model_a.predict_proba(features)
    │     └─ 釣魚風險分數 + High/Medium/Low 等級
    │
    ├─ Layer 3：model_b.predict(features) + model_c.predict(features_87d)
    │     └─ 釣魚類型（11 類）+ 嚴重程度
    │
    └─ Layer 4：HTTP POST http://lightrag:9621/query
          │  payload: { query, mode, top_k, ... }
          ▼
      [LightRAG]
          ├─ 向量搜索 / 圖譜遍歷（PostgreSQL pgvector + AGE）
          ├─ HTTP POST 100.66.161.11:11434/api/embed（Embedding）
          └─ HTTP POST 100.66.161.11:11434/api/chat（LLM 生成）
          ▼
      回傳 { response: "分析報告..." }
    │
    ▼ JSON 回應 → [Streamlit] 渲染結果頁面
```

## 2. 服務串接說明

### 2.1 phishrag-frontend → phishrag-backend

**協定**：HTTP REST（`requests` Python 函式庫）

**連線設定**（`dashboard.py`）：
```python
BACKEND_URL = os.getenv('BACKEND_URL', 'http://localhost:5000')
# Docker 環境：BACKEND_URL=http://phishrag-backend:5000
# 本機開發：預設 http://localhost:5000
```

**呼叫的端點**：

| 動作 | 方法 | URL | Timeout |
|------|------|-----|---------|
| 初始分析 | POST | `{BACKEND_URL}/api/v1/analyze` | 200s |
| 多輪追問 | POST | `{BACKEND_URL}/api/v1/chat` | 200s |
| 查詢知識圖譜 | POST | `{BACKEND_URL}/api/v1/query_graph` | 60s |

**進階查詢參數傳遞機制**（`dashboard.py: _rag_extra_params()`）：

```python
def _rag_extra_params() -> dict:
    raw = {
        'user_prompt':         st.session_state.get('user_prompt_input', ''),
        'top_k':               st.session_state.get('top_k_input'),
        'chunk_top_k':         st.session_state.get('chunk_top_k_input'),
        'max_entity_tokens':   st.session_state.get('max_entity_tokens_input'),
        'max_relation_tokens': st.session_state.get('max_relation_tokens_input'),
        'max_total_tokens':    st.session_state.get('max_total_tokens_input'),
    }
    return {k: v for k, v in raw.items() if v}  # 過濾 None/空值
```

這些參數以 `**kwargs` 展開後附加至每次 API 請求的 JSON body。

### 2.2 phishrag-backend → phishrag-lightrag

**協定**：HTTP REST（`requests` Python 函式庫）

**連線設定**（`app.py`）：
```python
_LIGHTRAG_BASE     = os.getenv('LIGHTRAG_URL', 'http://localhost:9621')
LIGHTRAG_QUERY_URL = f'{_LIGHTRAG_BASE}/query'       # LLM 生成
LIGHTRAG_DATA_URL  = f'{_LIGHTRAG_BASE}/query/data'  # 純圖譜資料
# Docker 環境：LIGHTRAG_URL=http://lightrag:9621
# 本機開發：預設 http://localhost:9621
```

**呼叫端點 1：`POST /query`（LLM 生成回應）**

```python
# app.py: query_lightrag()
payload = {
    "query": query_text,  # 組合後的完整查詢（郵件 + ML 結果 + 使用者指令）
    "mode":  rag_mode,    # hybrid / local / global / mix / naive / bypass
}
# 以下為選填，由前端進階參數傳入
if user_prompt:         payload["user_prompt"]          = user_prompt
if top_k:               payload["top_k"]                = top_k
if chunk_top_k:         payload["chunk_top_k"]          = chunk_top_k
if max_entity_tokens:   payload["max_entity_tokens"]    = max_entity_tokens
if max_relation_tokens: payload["max_relation_tokens"]  = max_relation_tokens
if max_total_tokens:    payload["max_total_tokens"]      = max_total_tokens
```

回傳：`{ "response": "LLM 生成的分析報告" }`

**呼叫端點 2：`POST /query/data`（圖譜原始資料，不呼叫 LLM）**

```python
# app.py: query_graph()
payload = { "query": query_text, "mode": rag_mode }
```

回傳：
```json
{
  "data": {
    "entities":      [{ "entity_name", "entity_type", "description", ... }],
    "relationships": [{ "src_id", "tgt_id", "keywords", "weight", "description" }],
    "chunks":        [{ "content", "file_path", ... }]
  },
  "metadata": { "processing_info": { ... } }
}
```

## 3. ML 流水線詳解（Layer 1–4）

### 3.1 Layer 1：特徵工程（76 維）

**程式碼位置**：`PhishRAG/app.py: extract_features()`

**特徵矩陣結構**：

#### 手工特徵（28 個）

| 群組 | 特徵名稱 | 說明 |
|------|---------|------|
| 文字長度（5） | `char_count`, `word_count`, `sentence_count`, `avg_word_length`, `avg_sentence_len` | 基礎統計量 |
| 標點風格（5） | `exclamation_count`, `question_count`, `uppercase_word_count`, `uppercase_ratio`, `special_char_count` | 緊迫感指標 |
| 詞彙分數（8） | `urgency_score`, `financial_score`, `authority_score`, `credential_score`, `threat_score`, `pii_request_score`, `brand_impersonation_score`, `social_engineering_score` | 各類惡意詞彙命中計數 |
| 合成分數（1） | `total_suspicious_score` | 8 個詞彙分數加權總和 |
| URL 特徵（5） | `has_url`, `url_count`, `has_suspicious_domain`, `url_text_mismatch`, `has_ip_url` | 連結惡意性 |
| 寄件者特徵（4） | `sender_domain_suspicious`, `display_name_mismatch`, `reply_to_mismatch`, `has_attachment` | 寄件人異常 |

#### TF-IDF 特徵（48 個）

從郵件 body 計算 48 個預定義高辨識力 bigram 的 TF-IDF 值，特徵名稱前綴為 `tfidf_`。詞彙表固定於 `PhishRAG_MLPipeline/tfidf_vocabulary.pkl`，確保推論一致性。

---

### 3.2 Layer 2：釣魚二元分類（Model A）

- **模型**：XGBoost（`model_a_binary.pkl`）
- **輸入**：76 維特徵向量
- **輸出**：
  - `is_phishing`：0（正常）/ 1（釣魚）
  - `risk_score`：釣魚機率（0.0–1.0）
  - `risk_level`：High（≥0.7）/ Medium（0.4–0.7）/ Low（<0.4）
  - `severity`：high / medium / low（轉換自 Model C 的嚴重程度）

---

### 3.3 Layer 3：釣魚分類（Model B + C）

**Model B**（`model_b_type.pkl`）：
- **輸入**：76 維特徵向量
- **輸出**：11 類釣魚類型，含信心度（`type_confidence`）
- **類別**：`authority_scam`, `credential_harvesting`, `financial_scam`, `generic_phishing`, `legitimate`, `romance_dating`, `social_engineering`, `social_engineering_advanced`, `tech_support`, `threats`, `urgency`

**Model C**（`model_c_severity.pkl`）：
- **輸入**：76 維特徵 + 11 維 Model B 預測機率 = **87 維**
- **輸出**：`low` / `medium` / `high` 嚴重程度

---

### 3.4 Layer 4：LightRAG 知識圖譜檢索

**查詢組合邏輯**（`app.py: _build_analysis_query()`）：

```
【PhishRAG 郵件威脅情資分析任務】

【待分析郵件內容】
{email_content}

【ML 流水線分析摘要（Layer 1–3）】
- 釣魚判定：{phish_label}
- 風險分數：{risk_score}
- 風險等級：{risk_level}
- 釣魚類型：{phishing_type}
- 分類信心度：{type_confidence}

【分析師指令】
{user_query}
```

這個組合查詢同時提供「郵件內容」和「ML 分析結果」給 LightRAG，讓知識圖譜能精確檢索相關的 MITRE ATT&CK 技術、威脅行為者和防禦建議。

多輪追問使用 `_build_chat_query()`，包含歷史對話記錄，繞過 Layer 1–3。

## 4. 知識圖譜視覺化原理

### 4.1 觸發流程

```
使用者點擊「🕸️ 查看知識圖譜」
    │
    ▼ [Streamlit: dashboard.py] call_query_graph()
    │  HTTP POST {BACKEND_URL}/api/v1/query_graph
    │  payload: { email_content, user_query, rag_mode, layer2_risk, layer3_classification }
    │
    ▼ [Flask: app.py] /api/v1/query_graph
    │  組合查詢（_build_analysis_query）
    │  HTTP POST http://lightrag:9621/query/data
    │  payload: { query, mode }
    │
    ▼ [LightRAG] /query/data（不呼叫 LLM）
    │  執行向量搜索 + 圖譜遍歷（PostgreSQL）
    │  回傳原始圖譜資料
    │
    ▼ [Flask] 解包 data.entities / data.relationships / data.chunks
    │  回傳 { status, entities, relationships, chunks, metadata }
    │
    ▼ [Streamlit: dashboard.py] _render_knowledge_graph(graph_data)
       儲存至 st.session_state.graph_data → st.rerun() 觸發重繪
```

for i, entity in enumerate(entities[:80]):   # 最多 80 個節點

### 5.3 RAG 模式說明

| 模式 | 檢索策略 | 速度 | 適用場景 |
|------|---------|------|---------|
| `hybrid` | 局部圖遍歷（鄰居擴展）+ 全局社群摘要 | 中 | 一般威脅分析（推薦）|
| `local` | 僅局部圖遍歷（以實體關係鏈） | 快 | 查詢特定 TTP、工具或行為者 |
| `global` | 僅全局社群摘要（跨實體統計） | 中慢 | 宏觀威脅態勢分析 |
| `mix` | 向量相似度 + 知識圖譜混合 | 中 | 語義接近但關係稀疏的查詢 |
| `naive` | 純向量相似度搜索（無圖譜） | 最快 | 快速初步查詢、圖譜資料不足時 |
| `bypass` | 直接交由 LLM 回答（不查圖譜） | 快 | 一般問答，不需知識庫支援 |

## 7. 開發與維運指引

### 7.1 本機開發（不使用 Docker 的 PhishRAG）

```powershell
# 仍需 PostgreSQL 和 LightRAG 在 Docker 中運行
cd "C:\Users\games\OneDrive\Desktop\PhishRAG資料夾\lightrag-package"
docker compose up -d   # 啟動 PostgreSQL

cd "..\LightRAG"
docker compose up -d   # 啟動 LightRAG

# 啟動 Flask（conda）
conda activate PhishRAG
cd "..\PhishRAG"
python app.py

# 啟動 Streamlit（conda，另開終端）
conda activate PhishRAG
cd "..\PhishRAG"
streamlit run dashboard.py
```

本機開發模式：`LIGHTRAG_URL` 和 `BACKEND_URL` 均使用預設值 `http://localhost:{port}`，無需設定環境變數。

---

### 7.2 新增知識庫文件

1. 開啟 LightRAG WebUI：`http://localhost:9621`
2. 進入「Documents」頁籤
3. 點擊「Upload」，選擇 PDF / Word / TXT 檔案
4. 系統自動執行：文件解析 → 實體抽取 → 關係建立 → 向量索引
5. 處理完成後知識圖譜即可搜索新增內容

> ⚠️ 大型文件處理需時，可在 WebUI「Documents」頁籤追蹤進度。

## 8. 版本紀錄

- 六種 RAG 模式切換（hybrid / local / global / mix / naive / bypass）
- Embedding 超時優化（TIMEOUT=180s, MAX_ASYNC=2, BATCH=4）

- 前端：Streamlit 1.30+
