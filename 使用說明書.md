# PhishRAG v1 使用說明書

**PhishRAG — 釣魚郵件威脅情資分析儀表板**

本系統整合三層 XGBoost 機器學習分類器與 LightRAG 知識圖譜，對可疑郵件進行自動化威脅情資分析，並支援多輪 AI 問答追問。

## 1. 系統需求

| 項目 | 最低需求 |
|------|---------|
| Docker Desktop | 4.x 以上（已啟用 WSL2 後端） |
| 記憶體 | 8 GB RAM（16 GB 建議） |
| 硬碟空間 | 10 GB 以上（含 Docker 映像） |
| 網路 | 可連線至 Tailscale（Ollama LLM/Embedding 服務） |
| 作業系統 | Windows 10/11、macOS、Linux |

> **重要**：LightRAG 的 LLM（`qwen3.5:9b`）與 Embedding（`qwen3-embedding:8b`）模型運行於遠端 Ollama 伺服器（`100.66.161.11:11434`，Tailscale 網路）。啟動前請確認 Tailscale 已連線。

## 2. 快速啟動

### 步驟 1：開啟終端機，進入專案根目錄

```powershell
cd "C:\Users\games\OneDrive\Desktop\PhishRAG資料夾"
```

### 步驟 2：啟動所有服務

```powershell
docker compose up -d
```

首次啟動需下載映像並建置 PhishRAG 映像，約需 3–10 分鐘。後續啟動使用快取，速度更快。

### 步驟 3：確認所有服務正常運行

```powershell
docker compose ps
```

當四個服務的狀態均顯示 `running (healthy)` 時，系統準備就緒。

| 容器名稱 | 說明 | 啟動時間 |
|---------|------|---------|
| `phishrag-postgres` | PostgreSQL 資料庫 | ~15 秒 |
| `phishrag-lightrag` | LightRAG 知識圖譜服務 | ~60 秒 |
| `phishrag-backend` | Flask 後端 API | ~30 秒 |
| `phishrag-frontend` | Streamlit 前端介面 | ~20 秒 |

> 各服務有健康檢查依賴，按順序等待前一服務就緒才啟動，總啟動時間約 2–3 分鐘。

### 步驟 4：開啟儀表板

在瀏覽器輸入：

```
http://localhost:8501
```

## 3. 功能操作說明

### 3.1 首頁（系統入口）

開啟儀表板後，您會看到標題列與說明文字，以及底部的郵件輸入區域。

---

### 3.2 分析頁面 — 貼入郵件並執行分析

**① 貼入待分析郵件**

在「✉️ 原始郵件內容 (Context)」文字框中，貼入完整的可疑郵件內文（包含主旨、寄件者、內文、連結等）。

**② 輸入初始分析指令（可選）**

在「初始分析指令」欄位輸入您的問題，例如：
- `請分析這封郵件使用了哪些社交工程手法？`
- `這封郵件與哪些 ATT&CK 技術有關？`

若留空，系統會使用預設指令：「請對此郵件進行完整的 PhishRAG 流水線威脅分析。」

**③ 選擇 LightRAG 檢索模式**

| 模式 | 說明 | 適用場景 |
|------|------|---------|
| `hybrid`（預設） | 局部圖遍歷 + 全局社群摘要混合 | 一般分析，推薦 |
| `local` | 以實體關係進行局部圖遍歷 | 查詢特定攻擊者或工具 |
| `global` | 以社群摘要進行全局圖推理 | 宏觀威脅情資概覽 |
| `mix` | 向量語義搜索 + 知識圖譜混合 | 概念性查詢 |
| `naive` | 純向量相似度搜索 | 快速初步查詢 |
| `bypass` | 跳過圖譜，直接問 LLM | 不需要知識庫輔助時 |

**④ 點擊「🚀 執行分析」**

系統依序執行四層流水線：
1. **Layer 1**：擷取 76 維技術特徵
2. **Layer 2**：XGBoost 二元分類（是否為釣魚？）
3. **Layer 3**：XGBoost 多分類（10 種釣魚類型）+ 嚴重程度評估
4. **Layer 4**：LightRAG 知識圖譜檢索 + ATT&CK 技術映射

分析時間依 LLM 回應速度而定，通常需 20–120 秒。

---

### 3.4 查看 Layer 1 技術特徵

點擊「🔍 檢視 Layer 1 技術特徵指標（76 維特徵向量）」展開器，可查看：
- **手工特徵**：文字長度、URL 數量、緊迫感詞彙分數、品牌假冒分數等 28 個特徵
- **TF-IDF 特徵**：48 個高辨識力詞組（bigram）的 TF-IDF 值

---

### 3.5 查看知識圖譜

點擊「🕸️ 查看知識圖譜」按鈕，系統會從 LightRAG 檢索與此郵件相關的實體和關係，以互動式網路圖呈現。

**圖譜操作**
- 滑鼠滾輪：縮放
- 拖曳節點：移動
- 懸停節點/邊：顯示詳細說明

**節點顏色說明**

| 顏色 | 實體類型 |
|------|---------|
| 🔴 紅色 | Technique / Attack / Tactic |
| 🔵 藍色 | Organization |
| 🟣 紫色 | Actor（威脅行為者）|
| 🟠 橘色 | Malware |
| 🟡 黃色 | Tool |
| 🟢 綠色 | Mitigation |
| 青色 | Indicator |
| 灰色 | 其他 |

圖譜下方另有三個資料頁籤：**Entities**（實體列表）、**Relationships**（關係列表）、**Chunks**（來源文字段落）。

---

### 3.6 多輪 AI 問答追問

分析完成後，郵件內容會鎖定為會話上下文，您可在「💬 資安 AI 助手」區域持續追問：

- `這封郵件中的惡意連結使用了哪些技術？`
- `如何防禦這類攻擊？`
- `有類似的歷史案例嗎？`

每次追問可切換不同的 RAG 模式，不會重複執行 ML 分析。

---

### 3.7 重置分析 / 分析新郵件

點擊「🔄 重置 / 分析新郵件」按鈕，清除當前會話所有狀態，回到初始輸入畫面。

## 4. 進階查詢參數

展開「⚙️ 進階查詢參數」可調整以下 LightRAG 檢索行為（留空使用伺服器預設值）：

| 參數 | 預設值 | 說明 |
|------|-------|------|
| Additional Output Prompt | 空 | 附加到 LightRAG 查詢的額外指令，例如「請用繁體中文回答並附上 ATT&CK 編號」 |
| KG TOP K | 40 | 從知識圖譜中檢索的實體/關係數量，越高越詳盡但越慢 |
| Chunk TOP K | 20 | 向量搜索返回的文字段落數量 |
| Max Entity Tokens | 6000 | 傳給 LLM 的實體描述最大 token 數 |
| Max Relation Tokens | 8000 | 傳給 LLM 的關係描述最大 token 數 |
| Max Total Tokens | 30000 | 傳給 LLM 的總上下文最大 token 數（含實體、關係、段落） |

## 5. 停止與重啟

### 停止所有服務

```powershell
docker compose down
```

> 此指令僅停止容器，不會刪除資料。PostgreSQL 資料和知識圖譜內容均保留於 Docker Volume。

### 完全清除（含資料）

```powershell
docker compose down -v
```

> ⚠️ **警告**：加上 `-v` 會刪除 `postgres_data` volume，清除所有知識圖譜資料，無法復原。

### 重啟單一服務

```powershell
docker compose restart <服務名稱>
# 例：
docker compose restart lightrag
docker compose restart phishrag-backend
```

### 查看服務日誌

```powershell
docker compose logs -f phishrag-backend   # Flask 後端日誌
docker compose logs -f lightrag           # LightRAG 日誌
docker compose logs -f phishrag-frontend  # Streamlit 日誌
docker compose logs -f postgres           # 資料庫日誌
```

## 6. 常見問題 FAQ

**Q：啟動後瀏覽器顯示「無法連線」？**

服務啟動需要 2–3 分鐘。執行 `docker compose ps` 確認四個服務均為 `healthy`，再開啟瀏覽器。

---

**Q：分析時顯示「無法連接後端服務」？**

1. 確認 Docker Desktop 正在運行
2. 執行 `docker compose ps` 查看服務狀態
3. 若 `phishrag-backend` 顯示 `unhealthy`，執行 `docker compose logs phishrag-backend` 查看錯誤原因

---

**Q：分析結果顯示「LightRAG 超時」或「Embedding timeout」？**

這是 Tailscale 網路延遲或 Ollama 負載過高導致的。解決方案：
1. 確認 Tailscale 已連線（`100.66.161.11` 可 ping 通）
2. 等待 1–2 分鐘後重試
3. 嘗試切換 RAG 模式為 `naive`（不使用圖譜，速度較快）

---

**Q：知識圖譜沒有顯示任何節點？**

請嘗試切換至 `local` 或 `global` 模式後重新點擊「查看知識圖譜」。`hybrid` 模式的圖譜覆蓋範圍最廣，但需要較多處理時間。

---

**Q：如何新增或更新知識庫內容？**

開啟 LightRAG WebUI（`http://localhost:9621`），在「Documents」頁面上傳 PDF、Word 或純文字檔案。系統會自動解析並建立知識圖譜。

---

**Q：修改了 `LightRAG/.env` 後需要重建映像嗎？**

不需要。`.env` 以 volume mount 方式掛入容器，修改後只需執行 `docker compose restart lightrag` 即可套用新設定。
